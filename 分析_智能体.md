# Crablet 智能体与认知系统深度分析报告

> 分析时间：2026-06-13  
> 分析范围：`crablet/src/{cognitive, agent, memory, tools, skills, workflow, knowledge, canvas}`  
> 分析目标：架构完整性、设计合理性、实现质量、扩展性、风险识别

---

## 1. 认知架构评估（System1/2/3/4）

### 1.1 整体架构设计

Crablet 实现了**四层认知架构**，对应 Kahneman 的 System 1/2 思维模型，并扩展至 System 3（群体智能）和 System 4（自进化）：

```
┌─────────────────────────────────────────────────────────┐
│ System 4 — Self-Evolving（自进化层）                     │
│  · 性能分析 · 技能发现 · 知识蒸馏 · 进化引擎              │
├─────────────────────────────────────────────────────────┤
│ System 3 — Swarm（群体智能层）                           │
│  · 多智能体协调 · 任务图分解 · 共识达成 · 辩论机制         │
├─────────────────────────────────────────────────────────┤
│ System 2 — Analytical（分析推理层）                      │
│  · ReAct 引擎 · 工具调用 · RAG · 树状思维 (ToT/MCTS)      │
│  · 中间件管道 · 分层推理 · 可观测性                       │
├─────────────────────────────────────────────────────────┤
│ System 1 — Intuitive（直觉响应层）                       │
│  · Trie 前缀匹配 · 模糊匹配 (Levenshtein) · 命令分发       │
└─────────────────────────────────────────────────────────┘
```

### 1.2 各层级评估

| 层级 | 设计合理性 | 实现质量 | 成熟度 | 关键问题 |
|------|-----------|---------|--------|----------|
| **System 1** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 高 | Trie 前缀匹配高效，但规则集仅 4 条硬编码命令，扩展性不足 |
| **System 2** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 高 | 中间件管道架构优秀，ReAct 可观测，分层推理(ToT/MCTS)配置丰富 |
| **System 3** | ⭐⭐⭐⭐ | ⭐⭐⭐ | 中 | Swarm 架构完整，但执行超时仅 2 分钟，Draft 模式功能单一 |
| **System 4** | ⭐⭐⭐⭐ | ⭐⭐⭐ | 中 | 概念完整，但 `process()` 仅处理状态查询，不直接处理用户任务 |

### 1.3 System 1 详细分析

- **Trie 前缀树**：字符级前缀匹配，时间复杂度 O(L)，对常见命令（hello/help/status/identity）实现亚毫秒响应
- **模糊匹配**：使用 `strsim::levenshtein` 距离，支持阈值自适应（短命令 0 容错，长命令 2 容错）
- **不足**：
  - 命令规则硬编码在 `System1::new()` 中，无法动态注册
  - 无上下文感知（`process` 签名接收 `context` 但忽略）
  - 多语言支持不完整（仅中英文 greeting）

### 1.4 System 2 详细分析（核心亮点）

**中间件管道（Middleware Pipeline）**是 System 2 的核心创新：

```rust
MiddlewarePipeline::new()
    .with_middleware(SafetyMiddleware)           // 安全审查
    .with_middleware(CostGuardMiddleware)        // 成本控制
    .with_middleware(SemanticCacheMiddleware)    // 语义缓存 (0.92 阈值)
    .with_middleware(RoutingMiddleware)          // 动态路由
    .with_middleware(PlanningMiddleware)        // 任务规划
    .with_middleware(RagMiddleware)             // 知识检索
    .with_middleware(SkillContextMiddleware)    // 技能上下文
```

- **分层推理**：通过复杂度分类器（Classifier）动态触发不同策略
  - 低复杂度 (<0.3) → 3 步 ReAct
  - 中等复杂度 → 5 步 ReAct
  - 高复杂度 (>0.8) → 10 步 ReAct
  - 超过 `deliberate_threshold` (0.58) → ToT (BFS/DFS)
  - 超过 `meta_threshold` (0.82) → MCTS Tree-of-Thoughts
- **策略切换**：当检测到重复工具调用或达到 max steps 时，自动切换策略并追加提示词
- **可观测性**：集成 `ObservabilityManager`（Tracer + BreakpointManager）

### 1.5 System 3 详细分析

- **Swarm 架构**：`SwarmCoordinator` → `SwarmExecutor` → `TaskGraph` 执行
- **注册的智能体**：Researcher、Coder、Analyst、Aggregator、Coordinator
- **TaskGraph 模板**：支持持久化模板、实例化、节点级更新/重试/恢复
- **不足**：
  - 超时硬编码为 120 秒，无动态调整
  - 仅 "draft " 和 "research " 前缀触发 System3，意图覆盖有限
  - 无负载均衡或背压机制

### 1.6 System 4 详细分析

- **四大引擎**：`PerformanceAnalyzer` → `SkillDiscoverer` → `KnowledgeDistiller` → `EvolutionEngine`
- **执行记录**：为 S1/S2/S3 提供统一的 `record_execution()` 接口，自动分类统计
- **知识蒸馏**：在后台将 System3 的成功执行轨迹转化为可复用知识
- **不足**：
  - `process()` 仅响应 `status`/`report`/`proposals` 等管理命令，不做任务处理
  - 进化引擎的 `start_evolution_loop()` 是空壳或待实现（未深入验证）
  - 改进提案（ImprovementProposal）需要人工审批，但审批后如何应用不明确

## 2. 记忆系统评估（三层记忆架构）

### 2.1 记忆层级总览

```
┌────────────────────────────────────────────┐
│ 工作记忆 (WorkingMemory) — 实时上下文窗口     │
│  VecDeque<Message> + Token 压缩 + 最近保留   │
├────────────────────────────────────────────┤
│ 情景记忆 (EpisodicMemory) — 持久对话历史       │
│  SQLite (WAL模式) + 后台批量写入队列          │
├────────────────────────────────────────────┤
│ 语义记忆 (SemanticMemory) — 知识图谱          │
│  SQLite 实体/关系表 + D3 JSON 导出            │
└────────────────────────────────────────────┘
```

### 2.2 Core Memory（MemGPT 风格）

- **三块结构**：Persona (500 chars) / Human (1000 chars) / Memory (2000 chars)
- **支持 CRUD**：`get`, `update`, `append`, `replace` 操作，自动截断到字符上限
- **版本控制**：`version` 字段用于冲突检测
- **设计亮点**：借鉴 MemGPT 的固定长度块设计，确保 LLM 始终能看到核心信息

### 2.3 Working Memory

- **双限制**：`capacity_messages`（软限制消息数）+ `max_tokens`（硬限制 Token 数）
- **压缩策略**：保留系统消息（index 0）+ 最近 4 条消息，丢弃中间消息
- **Token 估算**：使用 `tiktoken_rs::cl100k_base` 精确编码
- **不足**：
  - 压缩仅做丢弃，不做 LLM 总结（注释中留待 background consolidation 处理）
  - `compress_context()` 是同步方法，在 `add_message` 中同步阻塞

### 2.4 Episodic Memory

- **SQLite 持久化**：使用 `sqlx` + WAL 模式 + 64MB 缓存，配置合理
- **后台批量写入**：通过 `mpsc::channel` 实现异步批量写入，每批最多 20 条，单事务提交
- **Schema 迁移**：自动执行 `migrations/` 目录的迁移脚本
- **设计亮点**：生产级的数据库连接池配置（min=2, max=10, 5秒超时）

### 2.5 Semantic Memory / Knowledge Graph

- **SQLite 实现**：`entities` + `relations` 表，支持 D3 可视化导出
- **Trait 设计**：`KnowledgeGraph` trait 使用 `async_trait`，支持批量查询 `find_related_batch`
- **Neo4j 支持**：`#[cfg(feature = "knowledge")]` 下可选 Neo4j 后端（通过 `neo4rs`）
- **不足**：
  - 实体抽取依赖外部 LLM，未看到完全自动化的实体提取流水线
  - 关系类型单一，无本体论（Ontology）约束

### 2.6 Fusion Memory System（融合记忆 — 创新亮点）

**四层架构（OpenClaw 风格）**：

| 层级 | 名称 | 特性 | 对应文件 |
|------|------|------|----------|
| L1 | Session Layer | 实时会话上下文，DashMap 存储，1小时超时 | `layer_session.rs` (24K) |
| L2 | User Layer | 语义长期记忆，用户画像，可搜索 | `layer_user.rs` (47K) |
| L3 | Tools Layer | 动态工具记忆，工具调用历史 | `layer_tools.rs` (21K) |
| L4 | Soul Layer | 不可变核心身份，价值观，行为准则 | `layer_soul.rs` (15K) |
| - | Daily Logs | 追加式日志，按日期归档 | `daily_logs.rs` (19K) |
| - | Memory Weaver | 记忆整合、提取、合并、优化 | `weaver.rs` (15K) |

**设计亮点**：
- 会话结束时自动进行 L1→L2 记忆提取（`weaver.extract_from_session()`）
- 支持双向 Markdown 导出（`export_to_markdown`），便于人机协作
- 定时维护（`maintenance()`）：归档旧日志、合并重复记忆、清理过期会话、优化索引
- **适配器模式**：`FusionAdapter` 实现 Crablet/OpenClaw 配置互转

**风险点**：
- `layer_user.rs` 体积最大（47K），可能包含过度复杂逻辑，需进一步审查
- 用户隐私：L2 层记忆包含用户画像，无显式隐私擦除机制

## 3. 智能体协调评估（Swarm、分布式、MCP）

### 3.1 Swarm 系统

**架构组件**：

```
SwarmOrchestrator
├── SwarmCoordinator        (任务分解 + 模板管理 + 状态机)
│   ├── SwarmExecutor         (TaskGraph 执行 + 节点调度)
│   └── SwarmPersister      (SQLite 持久化)
└── Swarm                   (Agent 注册 + 消息路由 + 黑板)
    ├── SharedBlackboard    (共享状态)
    └── EventBus            (事件发布/订阅)
```

**TaskGraph 功能**：
- 节点级 CRUD：添加、更新、重试、恢复、取消
- 模板系统：保存成功执行图作为模板，支持实例化
- 执行状态：Pending → Running → Completed/Failed

**评估**：⭐⭐⭐⭐（架构完整，但执行细节待验证）

### 3.2 分布式系统

**模块**：`agent/distributed/`

- **后端抽象**：`InMemoryBackend` / `RedisHarnessBackend`（可插拔）
- **控制平面**：`HarnessControlPlane` + HTTP / In-Process / Noop 三种实现
- **领导者选举**：`LeaderElection` 模块
- **健康监控**：`HealthMonitor` + `NodeHealth` 状态
- **跨节点同步**：`DistributedHarnessManager` 管理 harness 状态同步

**评估**：⭐⭐⭐（接口设计良好，但 Redis 后端实现深度待验证）

### 3.3 MCP（Model Context Protocol）集成

**MCP 客户端实现**：`tools/mcp.rs`

- **完整 JSON-RPC 2.0**：支持 `initialize`, `tools/list`, `resources/list`, `prompts/list`, `tools/call`
- **异步 Reader 任务**：后台读取 stdout，通过 `DashMap` 分发响应到对应 oneshot channel
- **超时机制**：60 秒请求超时，自动清理 pending 请求
- **进程管理**：`kill_on_drop(true)` 确保 MCP 子进程随客户端释放而终止

**MCP 工具注册**：`System2::with_config()` 在启动时自动：
1. 连接配置中列出的 MCP 服务器
2. 注册 tools → `SkillRegistry` 作为 `SkillType::Mcp`
3. 注册 resources → 资源表
4. 注册 prompts → 提示词表

**评估**：⭐⭐⭐⭐⭐（业界领先的 MCP 实现，完整支持 Tools/Resources/Prompts 三要素）

## 4. 工具系统评估（扩展性、MCP 集成）

### 4.1 工具 Trait 设计

```rust
#[async_trait]
pub trait Tool: Send + Sync {
    fn name(&self) -> &str;
    fn description(&self) -> &str;
    fn parameters(&self) -> Value;
    async fn execute(&self, command: &str, args: Value) -> Result<String>;
}
```

- 简洁清晰，符合 OpenAPI 函数调用规范
- 支持 `parameters()` 返回 JSON Schema，便于 LLM 理解

### 4.2 原生工具矩阵

| 工具 | 功能 | 安全控制 | 状态 |
|------|------|----------|------|
| `bash` | 执行 Shell 命令 | SafetyOracle 严格模式 | 可用 |
| `file` | 文件读写 | SafetyOracle 路径检查 | 可用 |
| `http` | HTTP 请求 | 无额外限制 | 可用 |
| `search` | Web 搜索 | 可配置搜索提供商 | 可用 |
| `browser` | 浏览器自动化 | 无沙箱 | 可用 |
| `vision` | 图像分析 | 依赖 LLM 视觉能力 | 可用 |
| `mcp` | MCP 协议客户端 | 子进程隔离 | 优秀 |

### 4.3 扩展性分析

- **Plugin Trait**：`register_plugin()` 支持原生 Rust 插件注册
- **MCP 扩展**：通过外部进程动态扩展，无需重新编译
- **SkillManagerTool**：支持从目录加载技能包，运行时热加载（`SkillWatcher` 文件监控）
- **不足**：
  - 工具执行结果无统一结构化 schema（目前为 String）
  - 无工具执行权限粒度控制（如仅允许特定目录的 file 操作）

## 5. 技能系统评估（注册、发现、执行）

### 5.1 技能类型架构

```rust
pub enum SkillType {
    Local(Skill),           // 本地文件系统技能
    Mcp(SkillManifest, Arc<McpClient>, String),  // MCP 服务技能
    Plugin(SkillManifest, Arc<Box<dyn Plugin>>), // 原生 Rust 插件
    OpenClaw(Skill, String), // OpenClaw 提示词技能
}
```

**设计亮点**：
- 统一 `SkillManifest` 描述元数据（名称、版本、参数、依赖、权限、冲突）
- 支持 `triggers` 字段实现自动激活（条件触发）
- 版本管理（SemVer）、原子安装、依赖解析、语义匹配

### 5.2 技能注册中心

`SkillRegistry` 核心能力：
- 本地目录加载（`load_from_dir`）：扫描 `skill.yaml` / `skill.json`
- MCP 工具注册（`register_mcp_tool`）
- 插件注册（`register_plugin`）
- 远程注册表索引（GitHub 上的 `skill-registry` 索引）
- 备份/恢复（`backup` / `restore`）

### 5.3 技能执行链

```
SkillTriggerEngine → SkillOrchestrator → SkillExecutor
                                      ↓
                           CompositeSkill / SkillChain
                                      ↓
                           OpenClawEngine / 本地进程
```

- **Chain**：支持线性步骤链（`ChainStep` + `StepConnection`）
- **Composite**：支持并行/串行/条件组合（`CompositionType`）
- **OpenClaw**：基于提示词的技能执行，支持变量插值
- **DSL**：工作流定义语言 + 编译器

### 5.4 评估

| 维度 | 评分 | 说明 |
|------|------|------|
| 注册机制 | ⭐⭐⭐⭐⭐ | 四种类型统一注册，支持远程索引 |
| 发现机制 | ⭐⭐⭐⭐ | 语义搜索 + 混合匹配，支持注册表索引 |
| 执行机制 | ⭐⭐⭐⭐ | 链式/组合/DSL 丰富，但错误处理策略待完善 |
| 版本管理 | ⭐⭐⭐⭐ | SemVer 支持，原子安装，但无回滚机制 |
| 安全隔离 | ⭐⭐⭐ | 资源限制声明但未强制执行（`memory: "128MB"` 仅注释） |

## 6. 工作流系统评估（编排能力、可视化）

### 6.1 数据模型

工作流类型定义非常完善：

```rust
pub struct Workflow {
    pub id: String,
    pub name: String,
    pub nodes: Vec<WorkflowNode>,     // 节点（id, type, position, data）
    pub edges: Vec<WorkflowEdge>,     // 边（source, target, condition）
    pub variables: HashMap<String, Value>,
    pub version: i32,
    pub is_active: bool,
}
```

- 支持节点坐标（Position x/y），可直接映射到前端画布
- 支持条件边（`condition: Option<String>`）
- 支持节点端口（NodePort）类型定义

### 6.2 执行引擎

**WorkflowEngine** (`workflow/engine.rs`)：

```rust
pub async fn execute(&self, workflow_id: &str, req: ExecuteWorkflowRequest) -> WorkflowExecution {
    let id = next_id("exec");
    let started = now_rfc3339();
    // ... 直接返回 completed 状态，无实际节点执行
}
```

**⚠️ 关键缺陷**：`WorkflowEngine` 是**占位符/存根实现**：
- 没有真正的节点依赖解析
- 没有节点条件执行
- 没有边数据流传递
- 输入/输出映射未实现
- `NodeExecutorRegistry` 被标记为 `_executor_registry`（未使用）

**workflow/executor.rs**：
```rust
pub mod engine;
pub mod executor;
pub mod registry;
pub mod types;
```
其中 `executor.rs` 仅 136 bytes，说明执行器几乎为空。

### 6.3 可视化支持

`skills/visualization.rs`（27K）：
- `GraphExporter` 支持导出为 Mermaid、Graphviz、JSON 格式
- 节点颜色/图标分类（通过 `NodeTypeDefinition`）
- 但 WorkflowEngine 本身不生成可视化，仅依赖导出工具

**评估**：⭐⭐（数据模型优秀，执行引擎是空壳，需要大规模重构）

## 7. RAG/知识系统评估

### 7.1 EnhancedRAG 架构

```rust
pub struct EnhancedRAG {
    query_analyzer: QueryAnalyzer,       // 查询意图分析 + 实体抽取
    vector_store: Arc<VectorStore>,      // 向量存储（HNSW）
    knowledge_graph: SharedKnowledgeGraph, // 知识图谱（SQLite/Neo4j）
    config: RerankConfig,                // 重排序权重配置
    query_cache: moka::future::Cache<...>, // 查询缓存（5分钟 TTL）
}
```

### 7.2 多路召回（Multi-Route Retrieval）

```rust
pub struct MultiRouteResults {
    pub vector: Vec<RetrievedDocument>,   // 向量相似度
    pub keyword: Vec<RetrievedDocument>,  // 关键词匹配（简化 BM25）
    pub graph: Vec<RetrievedDocument>,    // 图谱实体关联
    pub expanded: Vec<RetrievedDocument>, // 查询扩展（子查询生成）
}
```

**融合重排序**（`fuse_and_rerank`）：
- 加权融合：vector (0.4) + keyword (0.2) + graph (0.2) + diversity (0.1) + recency (0.1)
- 去重（基于内容哈希）
- 查询缓存（moka 异步缓存）

### 7.3 GraphRAG

`knowledge/graph_rag.rs`（17K）：
- 实体抽取（EntityExtractorMode：Strict / Relaxed / Hybrid）
- 关系抽取与图谱构建
- 子图检索（基于实体查询相关子图）
- 与 VectorStore 协同使用

### 7.4 向量存储

`knowledge/vector_store.rs`（28K）：
- HNSW 索引实现（近似最近邻搜索）
- 支持多模态嵌入（文本 + 图像）
- SmartCache 层（LRU + 语义去重）

### 7.5 评估

| 维度 | 评分 | 说明 |
|------|------|------|
| 检索精度 | ⭐⭐⭐⭐ | 多路召回 + 重排序，查询分析器增强 |
| 召回率 | ⭐⭐⭐⭐ | 向量 + 关键词 + 图谱 + 扩展四路覆盖 |
| 性能 | ⭐⭐⭐⭐ | moka 缓存 + HNSW 索引 |
| 可解释性 | ⭐⭐⭐ | 返回 retrieval_type 和 score，但无详细归因 |
| 多模态 | ⭐⭐⭐ | 有 multimodal.rs 但集成深度待验证 |

## 8. 元认知系统评估（监控-反思-学习-优化闭环）

### 8.1 四阶段闭环架构

```
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│  Monitor │ ──→ │ Reflector│ ──→ │  Learner │ ──→ │ Optimizer│
│  (监控)   │     │  (反思)   │     │  (学习)   │     │  (优化)   │
└──────────┘     └──────────┘     └──────────┘     └──────────┘
     ↑                                               │
     └──────────────── 反馈调整 ─────────────────────┘
```

### 8.2 Monitor 阶段

**指标采集**：
- `ExecutionMetrics`：confidence, quality_score, resources, success, error
- `ResourceMetrics`：duration_ms, memory_bytes, cpu_ms, tokens_used
- `QualityMetrics`：completeness, accuracy, relevance, clarity（四级质量）
- `GlobalMetrics`：累计成功率、平均置信度、平均延迟、反馈历史

**设计亮点**：
- 支持任务级监控（start_execution / finish_execution）
- 全局指标滑动平均实时更新
- 反馈历史队列（`VecDeque<f32>`），支持用户显式反馈

### 8.3 Reflector 阶段

**问题诊断**：
- 问题分类：`ExecutionFailed` / `LowConfidence` / `ResourceExhaustion` / `PerformanceIssue` / `QualityIssue`
- 严重程度评估：基于规则的分级（如 confidence<0.3 → severity 0.8）
- **LLM 根因分析**：将指标喂给 LLM 生成根本原因（`analyze_root_cause`）
- **改进建议生成**：结合规则默认建议 + LLM 定制建议

### 8.4 Learner 阶段

**模式提取**：
- `TaskPattern`：按任务类别（coding/analysis/explanation/debugging/general）分类
- `ErrorPattern`：从诊断中提取错误模式存入知识库
- `StrategyPattern`：记录高置信度成功策略

**知识库结构**：
- 知识条目（LearnedKnowledge）+ 分类索引 + 关系图
- 模式去重：`find_similar_pattern` 基于类别匹配
- 成功率滑动平均更新

### 8.5 Optimizer 阶段

**策略优化**：
- `StrategyStats`：按策略统计使用率、成功率、平均置信度、平均延迟
- 策略选择：按成功率排序，选择最优策略（`get_best_strategy`）
- 分类策略映射：`enhanced` → coding/analysis, `fast` → explanation/general

### 8.6 闭环触发机制

```rust
pub async fn execute_with_meta(...) -> ExecutionResult {
    // 1. 监控开始
    monitor.start_execution(&task_id).await;
    // 2. 执行实际任务
    let result = executor(&request);
    // 3. 监控结束
    monitor.finish_execution(&task_id, &result).await;
    // 4. 如果失败或低置信度，触发反思
    if !result.success || result.confidence < learning_threshold {
        self.trigger_reflection(&request, &result).await;
    }
    result
}
```

**后台优化循环**：
```rust
pub async fn start_background_optimization(&self) -> Result<()> {
    tokio::spawn(async move {
        let mut ticker = tokio::time::interval(interval); // 默认 60 秒
        loop {
            ticker.tick().await;
            controller.run_optimization_cycle().await;
        }
    });
}
```

### 8.7 评估

| 维度 | 评分 | 说明 |
|------|------|------|
| 监控完整性 | ⭐⭐⭐⭐ | 多维度指标覆盖 |
| 反思深度 | ⭐⭐⭐⭐ | 规则+LLM 双驱动，根因分析有 LLM 参与 |
| 学习质量 | ⭐⭐⭐ | 模式提取基于简单规则，缺乏 LLM 辅助归纳 |
| 优化效果 | ⭐⭐ | 策略选择仅基于历史成功率，无 LLM 辅助优化 |
| 闭环自动化 | ⭐⭐⭐ | 自动触发，但优化结果仅记录未真正修改系统行为 |

### 8.8 关键不足

1. **Optimizer 是只读的**：`apply_improvements` 仅生成日志字符串，不修改任何系统配置、提示词或参数
2. **反射触发阈值固定**：`learning_threshold = 0.6` 硬编码，无自适应调整
3. **无负反馈循环**：成功策略会被强化，但失败策略不会被惩罚淘汰
4. **LLM 使用效率低**：根因分析调用 LLM，但结果未被有效利用（仅作为建议文本）

## 9. 优化建议清单

### 高优先级（P0）

1. **[工作流引擎重构]** `WorkflowEngine` 需要完整实现：
   - 拓扑排序执行（依赖解析）
   - 条件边执行（条件表达式求值）
   - 数据流映射（输入/输出变量绑定）
   - 节点级错误处理与重试

2. **[元认知优化落地]** 将 Optimizer 的优化结果真正应用到系统：
   - 动态调整 ReAct max_steps
   - 自动调整分层推理阈值（deliberate_threshold / meta_threshold）
   - 根据历史成功率优化 System 路由配置

3. **[System 1 扩展]** 实现动态命令注册：
   - 支持从配置文件或数据库加载命令规则
   - 支持上下文感知的命令分发（利用历史对话判断意图）

### 中优先级（P1）

4. **[工作记忆智能压缩]** 实现基于 LLM 的上下文总结：
   - 当 Token 超限且消息数 > capacity 时，用 LLM 压缩中间对话为摘要
   - 保留关键信息（代码、决策、用户偏好）

5. **[技能安全沙箱]** 实现资源限制强制执行：
   - 使用 `rlimit` 或容器化限制技能的 CPU/内存/网络
   - 实现权限模型的白名单/黑名单检查

6. **[RAG 查询分析增强]** 引入查询复杂度分析：
   - 根据查询类型动态调整召回策略权重
   - 引入 Causal Reasoning（`causal_reasoning.rs` 已存在）到查询扩展

7. **[Swarm 负载均衡]** 实现动态超时和并发控制：
   - 基于任务复杂度计算超时（而非固定 120s）
   - 实现 Agent 池化和任务队列（优先级队列）

### 低优先级（P2）

8. **[System 4 任务处理]** 允许 System 4 直接处理特定类型的优化请求：
   - 如 "分析我的使用模式并优化" 可触发 System 4 的完整进化周期
   - 而非仅返回状态报告

9. **[路由系统简化]** 合并冗余路由实现：
   - `adaptive_router`, `rl_router`, `speculative_router`, `meta_router` 功能重叠
   - 建议统一为 `MetaCognitiveRouter`（Contextual Bandit）+ 一个 Fallback

10. **[Memory 隐私增强]** 实现用户数据擦除（GDPR 合规）：
    - 支持按用户 ID 删除所有 L1/L2/L3 记忆
    - 支持会话级数据导出和删除

### 核心优势

1. **架构前瞻性**：四层认知 + 融合记忆 + MCP 集成，在开源 AI 框架中属于第一梯队
2. **MCP 实现**：业界最完整的 MCP 客户端之一，支持 Tools/Resources/Prompts 三要素
3. **可观测性**：System2 的 ReAct 引擎集成 Tracer 和 Breakpoint，支持调试和审计
4. **记忆分层**：从 MemGPT Core Memory 到 OpenClaw Fusion Memory，层次分明且可落地

### 核心短板

1. **工作流引擎**：严重缺失，当前是空壳，需要投入大量资源
2. **元认知落地**：反思和学习的结果停留在日志层面，未真正反馈到系统行为
3. **测试覆盖**：关键模块（Workflow、System3、System4、Knowledge）测试不足

```
Phase 1 (4 周): 工作流引擎重构 + 元认知优化落地
Phase 2 (4 周): 测试覆盖补齐 + System 1 动态扩展

```
