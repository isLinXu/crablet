## 1. Workspace 结构评估（Rust Workspace）

### 1.1 当前配置概览

```toml
[workspace]
members = ["crablet", "desktop"]
default-members = ["crablet"]
resolver = "2"

[workspace.dependencies]
tauri = { version = "2", features = ["tray-icon", "devtools"] }

[workspace.lints]
# ...
```

| 维度 | 评估 | 说明 |
|------|------|------|
| **成员结构** | ⚠️ 精简但合理 | 仅 2 个 crate：`crablet`（主后端）+ `desktop`（Tauri 桌面壳） |
| **默认成员** | ✅ 正确 | `default-members = ["crablet"]` 避免默认构建桌面端 |
| **Resolver** | ✅ 现代 | 使用 `resolver = "2"`，符合 Rust 1.80+ 生态 |
| **Workspace 依赖** | ✅ 统一 | `tauri` 版本在 workspace 中统一锁定，避免多版本冲突 |
| **Lints** | ✅ 良好 | 在 workspace 根定义 lint，但仅 `tauri-runtime` 锁定注释 |

### 1.2 关键发现

- **root Cargo.toml 极度精简**：仅包含 workspace 声明和 `tauri` 统一依赖，无 `package` 定义。这是合理的纯 workspace 根。
- **crablet/Cargo.toml 复杂度高**：146 行，24 个 feature 门控，大量 optional 依赖。`full` feature 组合了几乎所有功能模块。
- **desktop/Cargo.toml 简洁**：21 行，正确引用 `tauri = { workspace = true }`。
- **Rust MSRV**: `crablet` 声明 `rust-version = "1.80"`，CI 中单独验证 MSRV 编译 ✅

### 1.3 问题与风险

| 风险点 | 严重程度 | 说明 |
|--------|----------|------|
| `vision` feature 为空实现 | ⚠️ 低 | `vision = []` 为占位符，未声明任何依赖 |
| `fusion-tests-disabled` 存在 | ⚠️ 低 | 遗留 feature，可能已废弃 |
| `reqwest` 显式使用 `rustls-tls` | ✅ 好 | 避免了 OpenSSL 系统依赖，提升 CI/容器/本地一致性 |
| `openid` 使用 `rustls` + `http2` | ✅ 好 | 同样避免 native-tls |

### 2.1 Justfile（核心构建编排）

└── 通用 (dev, docker-build, up, clean, ci-smoke, ci-smoke-full, compose-validate)

| 任务 | 评估 | 说明 |

| `build-minimal` | ✅ | `--no-default-features --features web`，快速编译 |
| `build-full` | ✅ | `--release` 完整构建 |
| `test-web` / `test-doc` | ✅ | 与 CI 保持一致，使用 `--locked` |
| `lint` | ✅ | `fmt + clippy`，`-D warnings` 严格模式 |
| `audit` / `audit-ci` | ✅ | `cargo audit + cargo deny`，前端 `npm audit` |
| `coverage` | ✅ | `cargo llvm-cov` 生成 HTML/LCOV |
| `coverage-gate` | ✅ | 调用 `check_lcov_threshold.sh` 强制阈值 |
| `ci-smoke` / `ci-smoke-full` | ✅ | 本地模拟完整 CI，极佳开发体验 |
| `compose-validate` | ✅ | 带 CI-safe 环境变量验证 docker-compose 配置 |

| `dev` | ✅ | `serve-web --port 3000` 本地开发 |

### 2.2 Shell 脚本评估

| 脚本 | 评估 | 关键问题 |
|------|------|----------|
| `install.sh` | ✅ 设计良好 | Node.js 版本检查（20.19+/22.13+/24+）、自动生成 `.env`、sqlx-cli 安装、交互式 settings 调用 |
| `start.sh` | ⚠️ 可用但暴力 | 使用 `kill -9` 强制释放端口，对开发环境不友好；`cleanup` trap 中 `kill $(jobs -p)` 可能无法正确等待子进程 |
| `status.sh` | ✅ 实用 | 检查 Gateway (18789)、Web UI (3000)、MCP Servers，带颜色输出 |
| `uninstall.sh` | ⚠️ 有副作用 | `pkill -f "node"` 会杀死系统中所有 node 进程，非本项目专属 |
| `one-click.sh` | ✅ 协调脚本 | 统一入口，支持 `--start-only`、`--non-interactive`、`--uninstall` 等选项 |
| `settings.sh` | ✅ 交互体验好 | 支持 6 家 LLM 厂商预设、模型连通性验证、自动配置 `.env` |
| `scripts/build.sh` | ⚠️ 与 install.sh 不一致 | 默认 features 为 `knowledge,web`，而 `install.sh` 使用 `knowledge,auto-working,web` |
| `scripts/check_lcov_threshold.sh` | ✅ 严谨 | 使用 `set -euo pipefail`，awk 解析 LCOV，支持 Markdown 输出到 GitHub Step Summary |

### 2.3 构建流程不一致性

```
# install.sh 第 130 行
cargo install --path . --features knowledge,auto-working,web

# scripts/build.sh 第 17 行
FEATURES="knowledge,web"

# Justfile build-full
cargo build --manifest-path ./crablet/Cargo.toml --release
# 使用 crablet 默认 features: ["qdrant-support", "web"]
```

**问题**：三个入口的默认 feature 组合不一致。install.sh 多一个 `auto-working`，Justfile 使用 Cargo.toml 默认 features。这会导致不同构建路径产出的 binary 功能差异。

### 2.4 其他构建问题

| 问题 | 说明 |
|------|------|
| `docker-compose` 命令过时 | Justfile `up` 和 `clean` 使用 `docker-compose`（v1 已弃用），应改用 `docker compose`（v2） |
| 无 `Makefile` | 项目仅使用 Justfile，对未安装 `just` 的新用户不友好。建议保留一个简单 `Makefile` 作为 fallback |
| Node.js 版本约束仅在 install.sh | `frontend/package.json` 无 `engines` 字段，前端无法自我声明 Node 版本要求 |

## 3. Docker 容器化评估

### 3.1 Dockerfile 多阶段构建分析

```
Stage 1: frontend-builder  (node:26-alpine)      → 构建前端静态资源
Stage 2: chef              (cargo-chef:latest-rust-1.96) → 基础镜像
Stage 3: planner           (chef)               → cargo chef prepare 生成 recipe.json
Stage 4: builder           (chef)               → cargo chef cook 缓存依赖层
Stage 5: runtime           (debian:bookworm-slim) → 运行环境
```

| 维度 | 评估 | 说明 |
|------|------|------|
| **多阶段设计** | ✅ 优秀 | 使用 cargo-chef 分离依赖缓存层，显著加速重建 |
| **前端构建** | ✅ 正确 | Node 26 Alpine 构建前端，产物复制到 runtime |
| **非 root 用户** | ✅ 安全 | 创建 `crablet:crablet` 用户运行服务 |
| **健康检查** | ✅ 良好 | `curl -f http://localhost:3000/` 30s 间隔 |
| **运行时依赖** | ✅ 齐全 | ffmpeg、ca-certificates、libssl3、curl、python3 |
| **环境变量** | ✅ 合理 | `RUST_LOG=info`、`XDG_*` 路径设置 |
| **暴露端口** | ✅ 仅 3000 | Web UI 端口 |

### 3.2 Dockerfile 关键缺陷

| 缺陷 | 严重程度 | 说明 |
|------|----------|------|
| **未复制根 Cargo.toml** | 🔴 **高** | Planner 阶段仅 `COPY crablet/Cargo.toml Cargo.toml` 和 `crablet/src/`，但 **未复制根目录的 `Cargo.toml`**。由于 `crablet` 是 workspace 成员，且 `desktop` 引用 `tauri = { workspace = true }`，cargo chef 在 prepare 时可能需要 workspace 根配置。当前构建可能因缺失 workspace 根而隐式退化为独立 crate 构建，或导致版本解析不一致。 |
| **未复制 Cargo.lock** | 🔴 **高** | Planner 未复制 `Cargo.lock`（根目录或 crablet 下均无）。没有 lock 文件意味着 `cargo chef prepare` 基于 `Cargo.toml` 的 semver 解析，可能产生与开发环境不同的依赖版本，破坏可重复构建。 |
| **Rust 版本过新** | 🟡 中 | `cargo-chef:latest-rust-1.96` 使用 Rust 1.96，而项目 MSRV 为 1.80。虽然向下兼容，但 1.96 非常新，可能引入不稳定因素。建议固定到 `latest-rust-1.85` 或类似稳定版本。 |
| `python3-pip` 安装 | 🟡 中 | 运行时包含 pip，可能增加攻击面。若仅用于 MCP server 脚本执行，建议评估是否可移除或仅安装最小 python3。 |
| 前端 `node:26-alpine` | 🟡 中 | Node 26 尚未进入 LTS（预计 2026-10 成为 LTS），当前为最新版本。建议使用 `node:24-alpine` 与 `.nvmrc` 一致。 |
| 体积优化 | 🟡 中 | 未使用 `dpkg --add-architecture` 或精简包，但 debian:bookworm-slim 已较优。 |

### 3.3 docker-compose.yml 评估

**服务拓扑**（7 个服务）：

```
traefik          → 反向代理（443/80）
docker-proxy     → Docker Socket 安全代理
crablet          → 主 Web 服务（3000）
crablet-gateway  → API Gateway（18789）
neo4j            → 图数据库（7474/7687）
qdrant           → 向量数据库（6333/6334）
otel-collector   → OpenTelemetry 收集器（未暴露端口）
```

| 维度 | 评估 | 说明 |
|------|------|------|
| **网络隔离** | ✅ 良好 | `internal` 网络隔离内部服务，`proxy` 网络供 Traefik 发现 |
| **资源限制** | ⚠️ 部分 | `crablet` 限制 2G 内存 / 2 CPU，`crablet-gateway` 无限制 |
| **持久化卷** | ✅ 正确 | 数据、配置、Neo4j 日志、Qdrant 存储均挂载卷 |
| **重启策略** | ✅ 正确 | `unless-stopped` 适用于大多数服务 |
| **环境变量校验** | ✅ 良好 | `${NEO4J_PASSWORD:?...}` 强制非空 |
| **共享镜像** | ✅ 高效 | `crablet` 和 `crablet-gateway` 共用同一镜像，不同 `command` |

### 3.4 docker-compose.yml 问题

| 问题 | 严重程度 | 说明 |
|------|----------|------|
| **依赖不完整** | 🟡 中 | `crablet` 未 `depends_on` `qdrant`，但 `crablet-gateway` 却依赖了。`crablet` 也使用 `QDRANT_URL` 环境变量，缺失 `depends_on` 可能导致启动时序竞态。 |
| **无 healthcheck 依赖** | 🟡 中 | `depends_on` 仅保证容器启动顺序，不保证服务就绪。`crablet` 依赖 `neo4j` 和 `docker-proxy`，应使用 `depends_on.<service>.condition: service_healthy`（Compose v3+）。 |
| **Traefik 无服务发现** | 🟡 中 | 配置了 `traefik` 但 `crablet` / `crablet-gateway` 无 `traefik.*` 标签，Traefik 不会自动路由到它们。当前 Traefik 仅暴露了端口但无实际配置。 |
| **otel-collector 孤立** | 🟡 低 | `otel-collector` 存在但未暴露端口，且其他服务未配置 OTLP 端点。目前无服务向它发送遥测数据。 |
| `docker-proxy` 镜像旧 | 🟡 低 | `tecnativa/docker-socket-proxy:0.1.1` 发布于 2020 年，可能无 arm64 支持，且久未更新。建议评估 `ghcr.io/tecnativa/docker-socket-proxy:latest`。 |
| **无 `docker-compose.yml` 版本声明** | 🟡 低 | 无 `version: "3.8"` 等声明，依赖 Docker Compose 自动解析。这不是错误，但显式声明更清晰。 |

## 4. CI/CD 配置评估

### 4.1 Workflow 矩阵（6 个工作流）

| 工作流 | 触发条件 | 核心任务 | 评估 |
|--------|----------|----------|------|
| `ci.yml` | push/PR → main/develop | fmt, clippy, doc, check, test (web+doc), MSRV (1.80), build (ubuntu+macos), frontend lint/build/test | ✅ 全面 |
| `container.yml` | push/PR/tag → Dockerfile 变更 | compose 验证, image build, multi-arch publish (amd64+arm64) | ✅ 专业 |
| `security.yml` | push/PR + 每周一 02:17 | dependency-review, cargo audit, cargo deny, npm audit, Trivy FS 扫描 | ✅ 纵深防御 |

### 4.2 CI/CD 设计亮点

1. **路径过滤**：所有工作流使用 `paths` 过滤，避免无关变更触发全部任务 ✅
2. **并发控制**：`concurrency` 配置取消旧运行，节省资源 ✅
3. **权限最小化**：`permissions: contents: read` 为默认，仅发布 workflow 提升权限 ✅
4. **矩阵策略**：`fail-fast: false` 在测试矩阵中，确保所有 suite 都运行完 ✅
5. **缓存策略**：`Swatinem/rust-cache@v2` + `actions/cache@v4` 用于 advisory DB 和 tauri-cli ✅
6. **多平台发布**：`linux/amd64,linux/arm64` 容器镜像，QEMU + Buildx ✅
7. **MSRV 守护**：单独 job 使用 Rust 1.80.1 编译验证 ✅
8. **覆盖率门禁**：`check_lcov_threshold.sh` 在 CI 和 Justfile 中统一 80% 阈值 ✅

## 7. 部署流程评估（安装脚本、启动脚本）

### 7.1 安装流程

```
install.sh
├── [1/4] 检查 Rust 环境 (cargo --version)
├── [2/4] 检查 Node.js 环境 (node >= 20.19, 22.13, 24+)
├── [3/4] 构建后端
│     ├── 安装 sqlx-cli（全局 cargo install）
│     ├── 生成 .env（JWT_SECRET, DATABASE_URL, API keys）
│     ├── 调用 settings.sh（交互式 LLM 配置）
│     ├── sqlx database create + migrate run
│     └── cargo install --path . --features knowledge,auto-working,web
└── [4/4] 安装前端依赖 + 构建
```

### 7.2 部署问题

| 问题 | 严重程度 | 说明 |
|------|----------|------|
| `cargo install --path .` 污染全局 | 🟡 中 | install.sh 使用 `cargo install` 将 crablet 安装到 `~/.cargo/bin`，而非仅本地构建。这会导致版本冲突，且与 `start.sh` 寻找 `./target/release/crablet` 的本地 binary 不一致。 |
| `sqlx-cli` 全局安装 | 🟡 中 | 修改用户全局 cargo bin 目录。应使用 `cargo install --root ./.tools` 或检测是否已存在，避免覆盖。 |
| 无 `cargo install` 版本锁定 | 🟡 低 | `cargo install sqlx-cli` 未指定版本，可能安装最新版，与项目需求不匹配。 |
| `start.sh` binary 路径矛盾 | 🔴 **高** | `start.sh` 在 `crablet/` 目录下运行 binary，但先检查 `crablet/target/release/crablet`（相对项目根），然后执行时用 `$BINARY`（根路径）在 `cd crablet` 后执行。实际命令为 `CRABLET_SERVE_WEB_START_GATEWAY=false $BINARY serve-web...`，这会导致 binary 在错误 CWD 下运行（CWD 是 `crablet/` 而非根目录）。而 `cargo install` 安装的 binary 在 `~/.cargo/bin` 中。路径混乱。 |
| 无 systemd/Docker 生产部署脚本 | 🟡 中 | 仅有 `docker-compose.yml` 和 shell 脚本。缺少 systemd service 文件、Kubernetes manifests、或云原生部署指南。 |
| 无迁移回滚机制 | 🟡 低 | sqlx migrate 无 `revert` 脚本调用，数据库升级单向。 |
| `.env` 含空 API key 启动 | 🟡 低 | 默认 `.env` 中 API key 为空，应用可能启动但功能受限。建议在应用层做 key 缺失的优雅降级。 |

### 7.3 启动流程

`start.sh` 启动两个进程：
1. Web Server (`serve-web` on `${CRABLET_WEB_PORT:-3333}`)
2. Gateway (`gateway` on `${CRABLET_GATEWAY_PORT:-18790}`)

- 使用 `&` 后台进程 + `trap cleanup` 处理信号
- 但 `kill $(jobs -p)` 发送 SIGTERM 后不会等待 graceful shutdown
- 且 `kill -9` 在端口占用时过于暴力

### 8.1 当前支持矩阵

### 8.2 跨平台问题

| 问题 | 说明 |
|------|------|
| **桌面端仅 macOS arm64 + Windows x64** | 缺少 Linux 桌面（AppImage/flatpak/deb）和 macOS x64。Tauri 2 支持全平台，只需配置 CI 矩阵。 |
| **桌面 `sidecar` 硬编码平台** | `desktop-sidecar-copy` 仅复制 `aarch64-apple-darwin`，Windows CI 使用 `x86_64-pc-windows-msvc`。但本地开发时 `desktop-build` 无法在其他平台运行。 |

| `RUSTFLAGS` for Windows | CI 中 Windows 使用 `"-C target-feature=+crt-static"`，正确避免 MSVC 运行时依赖。 |
| `codesign --sign -` | macOS CI 使用 ad-hoc 签名，用户安装后需右键打开。未配置正式 Developer ID 签名。 |
| 无 musl 构建 | Linux 后端使用 `glibc`（Debian 基础镜像），若需 alpine/musl 部署需额外构建。 |

## 9. 优化建议清单（优先级排序）

### 🔴 P0 - 必须立即修复（阻塞构建/CI）

1. **修复 CI Action 版本**
   - 将所有 `actions/checkout@v6` → `v4`
   - 将 `actions/setup-node@v6` → `v4`
   - 将 `actions/upload-artifact@v7` → `v4`
   - 将 `codecov/codecov-action@v6` → `v4`
   - 将 `docker/setup-buildx-action@v4` → `v3`
   - 将 `docker/build-push-action@v7` → `v6`
   - 将 `docker/metadata-action@v6` → `v5`
   - 将 `docker/login-action@v4` → `v3`
   - 将 `docker/setup-qemu-action@v4` → `v3`

2. **修复前端 `package.json` 版本错误**
   - `typescript: ~6.0.2` → `~5.8.0`（或当前最新 5.x）
   - `@types/node: ^25.5.0` → `^24.0.0`（与 .nvmrc 一致）
   - 将 `@tauri-apps/cli-darwin-arm64` 从 `dependencies` 移到 `devDependencies` 或 `optionalDependencies`

3. **修复 Dockerfile 中 workspace 配置缺失**
   - Planner 阶段增加 `COPY Cargo.toml Cargo.toml`（根 workspace Cargo.toml）
   - Builder 阶段同样复制根 `Cargo.toml` 和 `Cargo.lock`

### 🟡 P1 - 高优先级（影响稳定性/一致性）

4. **统一 Node.js 版本策略**
   - 选项 A：全部统一为 Node 24（`.nvmrc`, CI, Dockerfile）
   - 选项 B：CI 中读取 `.nvmrc`：`node-version-file: '.nvmrc'`
   - Dockerfile 中 `node:26-alpine` → `node:24-alpine`

5. **统一默认 feature 组合**
   - `install.sh` 和 `scripts/build.sh` 默认使用相同的 feature set
   - 建议定义一个 `BUILD_FEATURES` 变量在 `settings.sh` 或 `.env` 中，供所有脚本引用

6. **修复 `start.sh` binary 路径问题**
   - 明确 binary 位置：若使用 `cargo install` 则应从 `~/.cargo/bin` 运行；若本地构建则应 `./target/release/crablet`
   - 建议移除 `cargo install --path .`，改用 `cargo build --release` + 本地运行

7. **修复 `uninstall.sh` 副作用**
   - `pkill -f "node"` → `pkill -f "vite"` 或更精确的模式，避免误杀全局 node 进程

8. **Dockerfile 中锁定 Cargo.lock**
   - 在 planner 和 builder 阶段复制 `Cargo.lock` 以确保可重复构建

9. **docker-compose.yml 添加服务健康依赖**
   - `crablet` 增加 `depends_on.qdrant`
   - 为 `neo4j` 添加 `healthcheck` 并在 `crablet` 中使用 `condition: service_healthy`

### 🟢 P2 - 中优先级（工程体验提升）

13. **添加 `engines` 到 `frontend/package.json`**

14. **Pre-commit 优化**

    - `markdownlint --fix` → `markdownlint --check`（在 pre-commit 中），CI 中保留 `--fix`

15. **添加 `cargo machete` 到 CI**
    - 定期检测未使用的依赖

## 10. 关键风险点总结

| 编号 | 风险 | 影响 | 缓解措施 |
|------|------|------|----------|
| R1 | **CI Action 版本不存在** | 所有 CI 工作流无法运行，阻塞发布 | 立即降级到 marketplace 可用版本 |
| R2 | **TypeScript 6.0 不存在** | 前端 `npm install` 失败，阻塞新开发者 | 降级到 `~5.8.0` |
| R3 | **Dockerfile 缺失 workspace 根 Cargo.toml** | 容器构建可能产生与本地不同的依赖版本 | 在 Dockerfile 中复制根 `Cargo.toml` 和 `Cargo.lock` |
| R4 | **桌面端依赖平台锁定** | Windows/Linux 开发者无法安装 desktop 依赖 | 将 `@tauri-apps/cli-darwin-arm64` 移至 `optionalDependencies` |
| R5 | **`start.sh` 路径不一致** | 本地启动失败或运行在错误 CWD | 统一使用本地 `target/release` 或全局 `~/.cargo/bin` |
| R6 | **Node 版本四处不一致** | 开发/CI/容器行为差异，难以调试 | 统一版本或让 CI 读取 `.nvmrc` |
| R7 | **uninstall.sh 误杀全局进程** | 开发者其他 Node 项目被意外终止 | 使用更精确的 `pgrep` 模式 |
| R8 | **docker-compose 依赖时序** | 启动时 qdrant 未就绪导致 crablet 启动失败 | 添加 `depends_on` + `condition: service_healthy` |
| R9 | **`cargo deny` 仅检查 `full` profile** | `web` 构建的依赖漏洞可能遗漏 | 在 CI 中对 `web` profile 也运行 `cargo deny` |
| R10 | **无 E2E 测试在 CI 中** | 发布前仅验证单元测试，集成问题可能漏到生产 | 将 E2E 测试加入 nightly CI 或 PR 合并前流程 |
