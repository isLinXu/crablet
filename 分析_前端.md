### 1.2 架构模式

- **SPA (Single Page Application)**：BrowserRouter + lazy-loaded routes
- **Feature-Based 目录结构**：按功能模块组织（chat / canvas / cognitive / dashboard / skills 等）
- **API Layer 分离**：`src/api/` (底层 client) + `src/services/` (业务封装)
- **Store 层分离**：`src/store/` 按领域划分（chat / model / canvas / theme / tokenStats）
- **Hooks 复用**：自定义 hooks 抽离业务逻辑（useStreamingChat, useAgentThinking, useApi, useKeyboard）

### 1.3 路由架构 (App.tsx)

```
/             → 重定向到 /chat
/chat         → ChatLayout（lazy + preload）
/dashboard    → Dashboard（lazy）
/canvas       → Canvas（lazy）
/skills       → SkillBrowser（lazy）
/activity     → ActivityCenter（lazy）
/mcp          → McpCenter（lazy）
/memory       → MemoryCenter（lazy）
/knowledge    → KnowledgeBase（lazy）
/settings     → SettingsPanel（lazy）
/observability→ TraceViewer（lazy）
```

**亮点**：
- 所有路由均采用 `React.lazy()` 懒加载
- App 组件预加载 `/chat` 和 `/dashboard`（通过 `useEffect`）
- Sidebar 的 `onMouseEnter` / `onFocus` / `onTouchStart` 触发预加载其他路由
- 全局 `ErrorBoundary` 包裹

### 2.3 ⚠️ 依赖冲突与潜在问题

1. **Peer Dependency 冲突（严重）**
   - `@vitejs/plugin-react@6.0.1` 要求 `vite@^8.0.0`，但实际安装 `vite@^7.3.1`
   - 导致 `npm install` 失败，必须使用 `--legacy-peer-deps` 强制安装
   - 进一步导致 Vitest 运行时 `@vitejs/plugin-react` 尝试导入 `vite/internal` 失败（`ERR_PACKAGE_PATH_NOT_EXPORTED`）
   - **建议**：升级 Vite 到 `^8.0.0` 或降级 `@vitejs/plugin-react` 到 `^5.x`

2. **TypeScript 6.0 弃用警告（中等）**
   - `tsconfig.app.json` 中 `baseUrl` 选项在 TS 6.0 中已弃用，将在 TS 7.0 移除
   - 当前构建报错：`error TS5101: Option 'baseUrl' is deprecated`
   - **建议**：添加 `"ignoreDeprecations": "6.0"` 临时规避，或迁移到 TS 5.x 稳定版

3. **冗余类型定义包（低）**
   - `@types/dompurify@3.2.0` 已标记为 deprecated（dompurify 自带类型）
   - `@types/tesseract.js@2.0.0` 已标记为 deprecated（tesseract.js 自带类型）
   - **建议**：移除这两个包

4. **潜在功能重叠（低）**
   - `pdf-parse` 和 `pdfjs-dist` 同时存在，可能冗余
   - `d3` 和 `chart.js` 同时存在，需确认 d3 是否被直接使用（Canvas 中可能通过 dagre 间接使用）

| **modelStore.ts** | ✅ `persist` (v1) | 模型提供商、路由决策、手动选择、会话绑定 | 包含 prompt 分类和模型推荐逻辑 |

### 4.2 chatStore 深度分析

**优势**：
- 使用 `persist` 中间件保存会话历史到 localStorage
- 实现了 `migrate` 函数处理版本升级（v3）
- 支持 `partialize` 只持久化必要字段（排除 `isConnected`、`isThinking` 等运行时状态）
- 包含 `materializeUnsavedSession` 和 `syncSessionSnapshot` 处理未保存会话迁移

**风险**：
- 376 行代码，session 状态机逻辑复杂，容易出 bug
- `sessionMessages` 和 `messages` 双轨存储，同步逻辑容易不一致
- 每次 `addMessage` 都触发完整 session 同步，大 session 可能有性能问题
- 缺少服务端同步机制，纯 localStorage 持久化在多设备场景下失效

### 4.3 modelStore 深度分析

**亮点**：
- 内置默认提供商（OpenAI gpt-4o-mini、Anthropic claude-sonnet、Google gemini-1.5-pro）
- 实现了 `classifyPrompt` 函数按关键词分类（image_gen / multimodal / coding / analysis / general）
- 支持按优先级、质量、速度三种策略选择模型
- 支持会话级别手动覆盖模型选择
- `syncFromBackend` 与后端配置同步

**问题**：
- 路由逻辑全部在前端，后端无法干预模型选择
- `apiKey` 以明文形式存储在 localStorage 中（通过 persist）
- 没有加密或安全存储机制

## 9. 总结

3. 补充测试覆盖率，建立 CI/CD 质量门禁
