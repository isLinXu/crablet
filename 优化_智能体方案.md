## 二、逐项优化方案

### 2.1 工作流引擎：空壳 → DAG 执行引擎

**当前问题** (`engine.rs` 第 24-55 行):
```rust
pub async fn execute(&self, workflow_id: &str, req: ExecuteWorkflowRequest) -> WorkflowExecution {
    let id = next_id("exec");
    // ... 直接构造 completed 状态，不执行任何节点
    let execution = WorkflowExecution {
        status: "completed".to_string(),
        node_executions: vec![/* single start node */],
        // ...
    };
    // ...
}
```

**问题根源**:
- `_executor_registry` 被声明但从未使用（下划线前缀）
- 不读取 `workflow.nodes` / `workflow.edges`
- 无拓扑排序、无依赖解析、无并行执行
- `stream_events` 也只是直接完成

**优化方案**:

已创建 **`engine_v2.rs`**（保留原 `engine.rs` 不破坏现有 API），核心改进：

1. **DAG 构建与校验**：从 `Workflow` 的 `nodes` + `edges` 构建有向图，使用 Kahn 算法检测环
2. **拓扑并行执行**：将当前入度为 0 的节点批量收集，用 `tokio::spawn` 并行执行
3. **依赖注入**：通过 `resolve_node_inputs` 将上游 `source_handle` 输出映射到下游 `target_handle` 输入
4. **事件流**：`stream_events` 细化为每个节点启动/完成/失败的事件
5. **失败策略**：当前实现为 fail-fast（检测到失败即终止），可扩展为 `continue_on_error` 模式
6. **持久化**：执行过程中写入 `executions` map，支持 `get_execution` / `cancel_execution`

**关键代码结构**（详见 `engine_v2.rs`）:
```rust
struct DagGraph {
    nodes: HashMap<String, WorkflowNode>,
    edges: Vec<WorkflowEdge>,
    adjacency: HashMap<String, Vec<String>>,
    in_degree: HashMap<String, usize>,
}

impl DagGraph {
    fn from_workflow(workflow: &Workflow) -> Result<Self, WorkflowEngineError> { ... }
}

// 核心执行循环
async fn execute_dag(...) -> Result<HashMap<String, Value>, WorkflowEngineError> {
    // 1. 拓扑排序 + 批量并行执行
    // 2. 依赖解析（source_handle -> target_handle）
    // 3. 节点级失败处理
}
```

**实施步骤**:
1. 扩展 `NodeExecutorRegistry` 添加 `execute(node, inputs) -> Result<HashMap<String, Value>>`
2. 将 `engine_v2.rs` 作为新模块引入，替换现有 `WorkflowEngine` 使用
3. 添加端到端测试：构建简单 DAG（A→B→C）验证并行度和输出传递

### 2.2 元认知优化器：只记日志 → 配置可落地

**当前问题** (`optimizer.rs` 第 150-202 行):
- `optimize_for_task_pattern` → 只返回描述字符串，不修改任何系统参数
- `apply_error_prevention` → 只记录 debug log，不注入重试/降级策略
- `reinforce_strategy` → 只更新内部 `strategy_scores`，不触发执行系统变更

**优化方案**:

已创建 **`optimizer_v2.rs`**，核心改进：

1. **引入配置管理接口** `ConfigManager`：提供 `get/set/set_batch/snapshot/rollback`
2. **引入策略执行接口** `StrategyExecutor`：提供 `apply_role_profile/throttle_tool/boost_strategy`
3. **实际配置变更**:
   - `timeout` 类错误 → 提升 `swarm.default_timeout_ms`，限流高频失败工具
   - `rate_limit` 类错误 → 降低 `swarm.max_concurrent`
   - `hallucination` 类错误 → 提升 `cognitive.validation_strictness` 到 `high`
   - 成功策略强化 → 不仅提升权重，还提升对应角色的并发限制
4. **原子批量应用**：所有变更先收集到 `Vec<ConfigChange>`，再调用 `set_batch` 原子写入
5. **快照与回滚**：每次优化前创建配置快照，支持 `rollback(version)` 回退

**关键示例**（错误预防实际落地）:

```rust
        }
        // ...
    }

```

**实施步骤**:
1. 为现有系统实现 `ConfigManager`（可基于现有配置系统或数据库）
2. 为 `SwarmExecutor` / `CapabilityRouter` 实现 `StrategyExecutor`
3. 将 `meta_controller` 中的 `Optimizer` 替换为 `OptimizerV2`
4. 添加评估闭环：每小时检查已应用改进的 `effectiveness`，低于阈值则回滚

### 2.3 System 1：硬编码 → 动态注册 + 上下文感知

**当前问题** (`system1.rs`):
- 所有 4 条命令（hello, identity, help, status）在 `new()` 中硬编码构造函数
- 无 `register`/`unregister` 接口
- `process` 的 `_context` 参数完全忽略（前缀下划线）
- 无权重、优先级、条件过滤机制

**优化方案**:

已创建 **`system1_dynamic.rs`**，核心改进：

1. **动态注册接口**:
   ```rust
   pub async fn register_rule(&self, rule: DynamicCommandRule);
   pub async fn unregister_rule(&self, rule_id: &str) -> bool;
   pub async fn set_rule_weight(&self, rule_id: &str, weight: f32) -> bool;
   ```
2. **上下文感知**:
   - 引入 `ContextSnapshot`（session_id, turn_count, last_intent, recent_topics）
   - `condition: Option<Fn(&ContextSnapshot) -> bool>` 支持条件触发（如 `turn_count <= 3` 的 onboarding 规则）
   - `context_tags` 实现话题匹配加分（`recent_topics` 命中 +0.15）
3. **Trie 多映射**：同一前缀可映射到多条规则，`search` 返回 `Vec<rule_id>`
4. **加权排序**：最终得分 = `score * rule.weight`，实现高权重规则优先

**新增规则示例**（上下文感知）:
```rust

    primary_command: "start".into(),
    weight: 1.2,
    context_tags: vec!["onboarding"],
    condition: Some(Arc::new(|ctx| ctx.turn_count <= 3)), // 仅新用户
    handler: Arc::new(|_, ctx| format!("欢迎新用户！第{}轮对话...", ctx.turn_count)),
}
```

**实施步骤**:
1. 将 `system1.rs` 中现有 `System1` 的测试保留，逐步迁移到 `System1Dynamic`
2. 在启动时调用 `register_rules(System1Dynamic::builtin_rules())`
3. 在网关/路由层注入 `session_id` 到 `Message`，使 `context.session_id` 可用
4. 从技能系统（skill registry）动态注入规则：技能安装时自动注册对应快捷命令

### 2.4 System 3 / Swarm 超时：硬编码 → 动态计算

**当前问题**:
- `system3.rs` 第 79 行：`timeout: Duration::from_secs(120)` 硬编码 2 分钟
- `coordinator/mod.rs` 第 65 行：仅 `goal.to_lowercase().starts_with("draft ")` 触发 draft 模式
- `executor.rs` `SwarmExecutionProfile`：`max_cpu_time_ms: 120_000`（researcher）固定
- `TaskNode` 默认 `timeout_ms` = 30000，无动态调整
- 无历史执行时间学习、无系统负载感知

**优化方案**:

已创建 **`swarm_dynamic_timeout.rs`**（可复用于 System 3 和 Swarm），核心改进：

1. **历史学习**：按 `(role, task_type)` 记录最近 100 次执行时间，计算 `avg` 和 `p95`
2. **系统负载感知**：注入 `SystemLoadSnapshot`（CPU、内存、LLM 队列深度、网络延迟）
3. **动态公式**:
   ```
   timeout = base × history_weight
           + complexity × 20s
           + base × load_score × 0.5
           + base × priority_factor
           + burst_bonus
           + risk_bonus（历史成功率低时 +30%）
   ```
   最终 clamp 在 `[5s, 300s]` 区间。
4. **System 3 适配**：`compute_system3_timeout(goal, context_len)` 自动从 goal 前缀推断 `task_type`（draft/research/coding/analysis），并计入上下文长度复杂度

**关键示例**:
```rust
// System 3 替换 120s 硬编码
let timeout = timeout_engine
    .compute_system3_timeout("Draft a report about AI", messages.len())
    .await;
match tokio::time::timeout(timeout, execution_future).await { ... }

// Swarm 任务替换 30s 默认
let timeout = timeout_engine
    .compute_timeout("coder", "coding", complexity=0.7, priority=128, is_burst=false)
    .await;
```

**实施步骤**:
1. 在 `System3::process` 中替换 `self.timeout` 为 `timeout_engine.compute_system3_timeout(...)`
2. 在 `SwarmExecutor::execute_graph` 中替换 `task_node.timeout_ms` 为动态计算值
3. 添加系统负载 observer（可用 `sysinfo` 或自定义指标）每 5s 调用 `update_system_load`
4. 在执行结束后调用 `record_execution` 闭环更新历史数据
5. 扩展 `decompose_and_execute` 的前缀触发：从硬编码 `"draft "` 改为基于复杂度评分的 router（`fusion_router` 已提供 `system3_threshold`）

### 2.5 工作记忆：无 TTL → 智能过期 + 重要性压缩

**当前问题** (`memory/working.rs`):
- 第 147 行：`is_expired` 存在但从未被外部调用
- 第 82-128 行：`compress_context` 仅简单 `pop_oldest_after_initial_message`，无重要性评估
- 无自动清理 task/background job 触发 TTL 检查

**优化方案**（建议修改 `working.rs`，无需新建文件）:

1. **引入 TTL 字段**:
   ```rust
   pub struct WorkingMemory {
       pub ttl: Duration,                    // 默认 30 分钟
       pub last_accessed: Instant,          // 已存在
       pub message_importance: HashMap<usize, f32>, // 每轮消息的重要性评分
       // ...
   }
   ```
2. **智能压缩策略**（替换 FIFO pop oldest）:
   - 保留系统消息（index 0）和最近 2 轮对话（`preserve_recent = 4`）
   - 对中间消息按重要性排序，优先删除 `importance < 0.3` 的消息
   - 重要性计算：基于 token 覆盖度（是否被后续消息引用）、实体提及次数、用户显式标记
   - 删除前尝试 `summarize`：将多条消息压缩为 1 条摘要消息（需异步 LLM 调用，可在 background 中完成）
3. **自动过期检查**：在 `memory/manager.rs` 或 session 层中每 60s 检查 `is_expired`，过期后自动 flush 到长期记忆并清空

**代码示例（压缩策略）**:
```rust
pub fn compress_context(&mut self) {
    // 1. 简单 TTL 检查（读 side 触发）
    if self.is_expired(self.ttl) {
        // 触发 flush 到长期记忆，由外部 manager 处理
        return;
    }

    // 2. 智能压缩：按重要性删除，而非 FIFO
    let preserve_recent = 4;
    while self.needs_compression() && self.history.len() > preserve_recent + 1 {
        // 找到中间区域中重要性最低的消息
        let mut min_idx = None;
        let mut min_score = f32::MAX;
        for (idx, _) in self.history.iter().enumerate().skip(1) {
            if idx >= self.history.len() - preserve_recent { break; }
            let score = *self.message_importance.get(&idx).unwrap_or(&0.5);
            if score < min_score {
                min_score = score;
                min_idx = Some(idx);
            }
        }
        if let Some(idx) = min_idx {
            let removed = self.history.remove(idx).unwrap();
            let tokens = self.estimate_message_tokens(&removed);
            // 更新后续索引的 importance map
            self.reindex_importance(idx);
            info!("Smart compression: removed low-importance message (score={:.2}, tokens={})", min_score, tokens);
        } else {
            break;
        }
    }
}
```

**实施步骤**:
1. 在 `WorkingMemory` 添加 `ttl: Duration` 和 `message_importance: HashMap<usize, f32>`
2. 在 `add_message` 时计算初始重要性（基于消息长度、是否包含关键词、是否被引用）
3. 在 `memory/manager.rs` 或 `Session` 中定期调用 `memory.is_expired()`，过期时触发 `consolidator` 归档
4. 引入后台任务（`background_thinker.rs` 已存在）定期执行 `compress_context` 的摘要版

### 2.6 工具系统：单调用 → 批量并行调用

**当前问题** (`tools/manager.rs`):
- `execute` 逐个线性遍历 `tools` Vec，逐个匹配、逐个执行
- `HttpPlugin::read_url` 每次调用新建 `reqwest::Client`（第 46 行），无连接复用
- 无 `execute_batch` 接口，无法合并多个 HTTP 请求或 MCP 调用

**优化方案**（建议修改 `manager.rs` + `http.rs`）:

1. **批量执行接口**:
   ```rust
   pub struct ToolCall {
       pub tool_name: String,
       pub args: Value,
       pub id: String, // 用于结果关联
   }

   pub struct ToolResult {
       pub id: String,
       pub result: Result<String>,
   }

   impl ToolManager {
       pub async fn execute_batch(&self, calls: Vec<ToolCall>) -> Vec<ToolResult> {
           let futures: Vec<_> = calls.into_iter().map(|call| {
               let tool_name = call.tool_name.clone();
               async move {
                   let result = self.execute(&call.tool_name, call.args).await;
                   ToolResult { id: call.id, result }
               }
           }).collect();
           futures::future::join_all(futures).await
       }
   }
   ```
2. **HTTP 连接复用**: `HttpTool` 使用 `lazy_static` 或 `Arc<Client>` 共享 client，启用连接池
3. **MCP 批量**: 对于 MCP 协议，添加 `mcp_batch_call` 将多个 tool call 打包为单个 JSON-RPC batch 请求
4. **智能合并**: 同类工具（如多个 `read_url`）可合并为并发 HTTP 请求，而非串行

**实施步骤**:
1. 在 `ToolManager` 添加 `execute_batch` 方法
2. 在 `HttpTool` 中引入 `Arc<reqwest::Client>` 复用连接
3. 在 `mcp.rs` 或 `mcp_plugins.rs` 中实现 JSON-RPC batch 请求
4. 在 `enhanced_react.rs` 或 agent 执行循环中，将多个工具调用收集后统一 `execute_batch` 而非逐个调用

## 四、代码质量与风险

| 风险 | 缓解措施 |
|------|----------|
| `engine_v2` 的 `DagGraph` 可能有未覆盖的 edge case | 补充 cycle detection + deadlock 测试；先灰度运行 |
| `optimizer_v2` 配置回滚依赖外部 `ConfigManager` 实现 | 先提供 `InMemoryConfigManager` 做 MVP，逐步对接数据库 |
| `System1Dynamic` 动态规则增加内存占用 | 限制总规则数（如 1000 条），引入 LRU 过期 |
| `DynamicTimeout` 历史数据膨胀 | 固定窗口 100 条，定期聚合为统计直方图 |
| TTL 清理可能导致 session 状态丢失 | 过期前强制 `consolidator` 归档，仅清除工作记忆，不删除数据库记录 |
